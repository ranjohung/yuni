## 1. 文档概述

### 1.1 产品定位

「与你」是一款AI驱动的数字人社交模拟训练App，通过高拟真数字人实时互动，结合认知行为疗法（CBT）、依恋理论、非暴力沟通（NVC）等心理学框架，为用户提供零压力的社交场景模拟训练与情感陪伴。

### 1.2 一句话价值主张

> **"先和TA练一遍，再去面对真实世界。"**

### 1.3 核心差异化

| 维度 | 传统AI聊天 | 「与你」 |
|------|-----------|---------|
| 目标 | 陪伴/闲聊 | 社交技能训练 |
| 心理学框架 | 无 | CBT + 依恋理论 + NVC |
| 训练结构 | 无 | 六阶段社交生命周期 |
| 反馈机制 | 无 | 即时反馈 + 评估报告 + 情绪日记 |
| 成长闭环 | 无 | 时空穿梭 + 学习卡片 + 晚安计划 + 每周报告 |
| 形象联动 | 无 | 与悦己颜值社联动 |
| 留存体系 | 无 | 每日签到 + 晚安计划 + 里程碑 + 关系危机 |

### 1.4 技术目标

| 指标 | MVP目标 | 正式版目标 |
|------|---------|-----------|
| 平台支持 | iOS / Android | iOS / Android |
| AI推理响应 | < 3秒（流式输出） | < 2秒 |
| 数字人渲染 | 2D动态立绘 | 3D ≥ 30fps |
| 语音通话延迟 | 1-2秒（MVP可选功能） | < 500ms |
| 并发承载 | 1万 DAU | 10万 DAU |

## 2. 产品定位与用户画像

### 2.1 核心目标用户

18-35岁、有社交焦虑或情感陪伴需求的都市年轻人（女性占比预计65-70%）。

### 2.2 四类用户画像

#### 第一类：社交焦虑人群（核心用户）
| 维度 | 描述 |
|------|------|
| 年龄 | 22-30岁 |
| 特征 | 害怕社交场合、不会聊天、容易冷场、被拒绝后过度自责 |
| 痛点 | 想练习社交但没有安全的环境 |
| 需求 | 零压力的社交训练场 |

#### 第二类：职场新人（高频用户）
| 维度 | 描述 |
|------|------|
| 年龄 | 22-28岁 |
| 特征 | 刚入职、需要面试/汇报/向上沟通 |
| 痛点 | 紧张、表达混乱、不敢开口 |
| 需求 | 职场场景模拟演练 |

#### 第三类：情感陪伴需求者
| 维度 | 描述 |
|------|------|
| 年龄 | 18-35岁 |
| 特征 | 孤独、寻求陪伴、缺乏倾诉对象 |
| 痛点 | 有情绪需要释放、渴望被倾听 |
| 需求 | 随时在线的情绪支持 |

#### 第四类：自我提升爱好者
| 维度 | 描述 |
|------|------|
| 年龄 | 25-35岁 |
| 特征 | 关注个人成长、愿意为自我提升付费 |
| 痛点 | 理论知识多但缺乏练习机会 |
| 需求 | 可量化的社交能力训练+每周报告 |

### 2.3 用户兴趣偏好
- 关注个人成长、心理健康
- 喜欢养成类游戏/互动叙事
- 为情绪价值付费意愿高
- 对AI/科技产品接受度高

## 3. 核心技术架构

### 3.1 技术选型总表

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 客户端框架 | Flutter | 跨端开发，iOS/Android一套代码 |
| 数字人渲染(MVP) | Spine 2D骨骼动画 | 免费，轻量，快速上线 |
| 数字人渲染(V1.1+) | 魔珐星云SDK / Three.js | 高拟真3D |
| 实时音视频 | 声网Agora RTC | 全双工语音通话 |
| LLM对话引擎(主力) | DeepSeek-Chat | 效果最优，按token付费 |
| LLM对话引擎(兜底) | Ollama + Qwen2.5:14B | 免费，自动降级 |
| TTS语音合成 | Microsoft Azure TTS | 多音色，晚安计划语音 |
| 情绪识别 | DeepSeek文本分析 | MVP以文本为主 |
| 长期记忆 | MySQL JSON字段 | MVP不上向量数据库 |
| 后端框架 | Node.js + Express | 轻量快速迭代 |
| 数据库 | MySQL 9.6 | 业务数据持久化 |
| 缓存 | Redis | Session、热点数据 |
| 对象存储 | 阿里云OSS | 用户照片、素材、晚安语音 |
| 定时任务 | node-cron | 晚安计划推送、朋友圈生成、每周报告 |

### 3.2 LLM混合引擎架构

```
用户输入 → LLM路由器
  ├─ 体验用户(1次) → DeepSeek
  ├─ 免费用户 → Ollama + 10%偶遇DeepSeek
  ├─ 日卡/周卡/月卡 → DeepSeek
  └─ 年卡 → DeepSeek高优先级
       → 健康检查失败 → 自动降级Ollama
       → 输出给用户
```

### 3.3 成本控制策略

| 策略 | 措施 | 节省 |
|------|------|------|
| LLM混合路由 | DeepSeek+Ollama | 减少API约80% |
| 内容预编译 | 预设对话树 | 60%走预设 |
| 本地优先 | 好感度客户端计算 | 服务器成本 |
| 免费素材优先 | Spine 2D+无版权 | SDK授权费 |
| MySQL+JSON | MVP不上向量库 | 服务器成本 |

### 3.4 情绪识别服务完整规格

#### 3.4.1 三层分析引擎

**第一层：文本情感分析**

| 类别 | 关键词 | 权重 |
|------|--------|------|
| 焦虑 | 担心/害怕/紧张/焦虑/慌/压力大 | 0.8 |
| 悲伤 | 难过/伤心/失落/绝望/没希望/想哭 | 0.8 |
| 愤怒 | 生气/气死了/烦/受不了/凭什么 | 0.7 |
| 恐惧 | 怕/不敢/恐惧/吓人/完了/死定了 | 0.8 |
| 委屈 | 委屈/冤枉/不公平/凭什么是我 | 0.7 |
| 开心 | 开心/高兴/好棒/太好了/幸福 | 0.6 |
| 平静 | 还好/还行/一般/没事/正常 | 0.3 |

**融合策略**：最终文本情感 = 0.4 × 关键词匹配 + 0.6 × BERT模型输出

**第二层：行为分析**

| 行为 | 分析维度 | 推断情绪 |
|------|---------|----------|
| 登录时间 | 深夜（>23:00） | 焦虑/失眠 |
| 对话频率 | 短时高频 | 焦虑/急切 |
| 对话长度 | <5字 | 低落/回避 |
| 训练行为 | 回避挑战 | 恐惧/焦虑 |

**第三层：语音情绪分析（月卡+）**

| 特征 | 对应情绪 |
|------|----------|
| 语速偏快 | 兴奋/焦虑 |
| 音量偏低 | 悲伤/疲惫 |
| 音调偏高 | 兴奋/焦虑 |
| 停顿频繁 | 紧张/不确定 |

#### 3.4.2 融合决策

**多模态融合公式**：
```
最终情绪强度 = 0.5 × 文本 + 0.3 × 语音（如有） + 0.2 × 行为
```

**决策规则**：

| 条件 | 触发模式 |
|------|----------|
| 强度 > 0.7 且无具体问题 | 树洞模式 |
| 强度 < 0.4 且有具体问题 | 建议模式 |
| 强度 > 0.6 且有具体问题 | 混合模式 |
| 强度 > 0.8 | 强制树洞模式 |

#### 3.4.3 成本优化方案

| 用户等级 | 文本分析 | 语音分析 | 行为分析 |
|----------|----------|----------|----------|
| 体验版 | 关键词匹配 | ❌ | 基础 |
| 免费 | 关键词匹配 | ❌ | 基础 |
| 周卡 | 关键词+BERT | ❌ | 完整 |
| 月卡 | 关键词+BERT | ✅ | 完整 |
| 年卡 | 关键词+BERT | ✅ | 完整 |

#### 3.4.4 API接口

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 文本情绪分析 | POST | /api/v1/emotion/text | 分析文本情绪 |
| 语音情绪分析 | POST | /api/v1/emotion/voice | 分析语音情绪 |
| 综合情绪分析 | POST | /api/v1/emotion/fusion | 多模态融合分析 |
| 获取情绪历史 | GET | /api/v1/emotion/history | 获取情绪变化趋势 |

#### 3.4.5 数据库表

```sql
CREATE TABLE emotion_records (
    id VARCHAR(20) PRIMARY KEY,
    user_id BIGINT NOT NULL,
    session_type VARCHAR(20),
    source_type VARCHAR(20),
    raw_text TEXT,
    emotion_primary VARCHAR(20),
    emotion_secondary JSON,
    intensity DECIMAL(3,2),
    polarity VARCHAR(10),
    confidence DECIMAL(3,2),
    trigger_mode VARCHAR(20),
    features JSON,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_id (user_id),
    INDEX idx_emotion (emotion_primary)
);
```

#### 3.4.6 健康检查机制（问题20补充）

**DeepSeek API健康检查策略**：

| 检查项 | 配置值 | 说明 |
|--------|--------|------|
| 检查间隔 | 5分钟 | 每5分钟对DeepSeek API进行一次ping请求 |
| 失败阈值 | 连续3次 | 连续失败3次（15分钟）触发降级 |
| 恢复尝试间隔 | 10分钟 | 降级后每10分钟尝试恢复一次 |

**降级期间用户提示**：对话界面显示“✨ 当前使用备用模型，对话体验可能略有不同”

**配置项**（config.js）：
```typescript
{
  DEEPSEEK_API_HEALTH_CHECK_INTERVAL: 300000,
  DEEPSEEK_API_FAIL_THRESHOLD: 3,
  DEEPSEEK_API_RECOVERY_INTERVAL: 600000
}
```

## 15. 心理健康教育模块

### 15.1 情感陪伴模式

「与你」的情感陪伴系统具备三种响应模式，根据用户情绪强度和是否有具体问题自动切换：

#### 15.1.1 三种模式定义

**模式一：树洞模式（先接住情绪）**

| 触发条件 | 行为表现 |
|----------|----------|
| 情绪强度 > 0.7 且无具体问题 | 伴侣先倾听、不急着给建议 |
| 用户说"我好累""好烦"类表达 | 伴侣先共情，不急于解决 |

**话术示例：**
- "听起来你真的很难过，我在这里陪你"
- "想多说点吗？我在这里"
- "换谁遇到这种情况都会不好受的"

**"无具体问题"判断逻辑**（问题12补充）：

| 判断条件 | 规则 | 示例 |
|----------|------|------|
| 关键词检测 | 不包含"怎么办、如何、怎么、为什么、原因、解决、方法"等问题类关键词 | 包含→有问题，不包含→无问题 |
| 场景检测 | 不包含具体人物/事件/场景（如"今天开会"、"我朋友"、"领导"等） | 包含→有问题，不包含→无问题 |
| 最终判定 | 两个条件都满足时，判定为"无具体问题"，进入树洞模式 | - |

**"倾听"行为技术实现**（问题22补充）：

| 实现方式 | 规则 | 说明 |
|----------|------|------|
| System Prompt | 附加"你当前处于树洞模式，请以倾听为主，不要主动提供解决方案或建议" | 引导AI角色定位 |
| 回复长度 | 控制在20-50字之间 | 简短回应，不喧宾夺主 |
| 回复风格 | 以共情句式为主（"我明白了"、"我能理解你的感受"、"我在这里陪你"等） | 温暖、支持性语气 |

**模式二：建议模式（给选项不给指令）**

| 触发条件 | 行为表现 |
|----------|----------|
| 情绪强度 < 0.4 且有具体问题 | 伴侣提供2-3个结构化选项 |
| 用户主动问"怎么办" | 伴侣先共情确认，再给建议 |

**话术示例：**
- "我理解你现在的情况确实很棘手，我想了几个方向，你看哪个更适合你..."
- "你觉得哪个思路更符合你的情况？"
- "如果选第一个，具体可以这样开始..."

**模式三：混合模式（先树洞后建议）**

| 触发条件 | 行为表现 |
|----------|----------|
| 情绪强度 > 0.6 且有具体问题 | 先共情接住，情绪缓解后再给建议 |

**流程示例：**
1. 第一轮："听起来真的很委屈，我陪你待一会儿"
2. 第二轮："现在感觉好些了吗？要不要一起理一理？"
3. 第三轮："我觉得可以试试这样做...你觉得呢？"

**强制树洞模式**：当情绪强度 > 0.8时，无论是否有具体问题，强制进入树洞模式，先安抚情绪。

#### 15.1.2 模式切换规则

| 当前模式 | 切换条件 | 目标模式 |
|----------|----------|----------|
| 树洞模式 | 用户明确说"那该怎么办"类 | 建议模式 |
| 建议模式 | 用户情绪强度上升（>0.6） | 混合模式 |
| 混合模式 | 用户情绪缓解（<0.4） | 建议模式 |
| 任意模式 | 用户手动切换（输入"我只想倾诉"/"给我建议"） | 对应用户选择 |

#### 15.1.3 与情绪识别服务联动

情绪识别服务（见第3.4节）计算出情绪强度后，传递给对话引擎，对话引擎根据规则选择响应模式：

```
用户输入 → 情绪识别 → 强度计算 → 模式决策 → AI生成回复
```

#### 15.3.2 5步填写流程

**第1步：情境**
- 输入字段：文本输入框
- 提示文案："尽量客观描述，不要加入评价"
- 示例："今天开会时，我提出的方案被领导直接否定"

**第2步：想法**
- 输入字段：文本输入框
- AI自动检测并展示认知扭曲类型
- 提示文案："灾难化思维是指把事情想象得比实际更糟"

**第3步：情绪**
- 输入字段：多选情绪+强度滑块（1-10分）

**第4步：证据**
- 支持证据：文本输入框，写下所有支持这个想法的客观事实
- 反对证据：文本输入框，写下所有不支持这个想法的客观事实

**第5步：替代想法**
- 输入字段：文本输入框，写下更合理的想法
- AI基于用户填写的证据自动生成建议
- 重新评估情绪强度（1-10分）

#### 15.4.2 4步填写流程

**第1步：观察**
- 输入字段：文本输入框
- 要求：描述客观事实（不带评价）
- 好的观察："当你提高音量说话时"
- 避免评价："当你态度很差时"

**第2步：感受**
- 输入字段：下拉选择
- 可选感受：受伤/生气/难过/焦虑/失望/挫败/委屈/害怕/其他
- 好的感受："我感到受伤"
- 避免想法："我觉得你不对"

**第3步：需要**
- 输入字段：下拉选择
- 可选需要：尊重/被倾听/理解/信任/公平/支持/安全感/合作/其他
- 好的需要："因为我需要被尊重"

**第4步：请求**
- 输入字段：文本输入框
- 要求：具体、可行
- 好的请求："你愿意下次先用平和的语气和我说吗？"
- 避免模糊："你愿意对我好一点吗？"

#### 15.4.3 完整句子预览

```
当你{观察}（观察），我感到{感受}（感受），因为我需要{需要}（需要），你愿意{请求}吗？（请求）
```

**预览区域UI布局：**
- 顶部显示完整句子，每部分用不同颜色标注（观察=蓝色，感受=红色，需要=绿色，请求=橙色）
- 句子下方显示"发送到对话"按钮
- 点击按钮后，完整句子作为用户消息发送到对话界面

#### 15.4.4 发送到对话的交互流程

```
用户完成4步填写 → 预览完整句子 → 点击[发送到对话] → 句子作为用户消息发送 → AI伴侣以NVC方式回应 → 好感度+15
```

**前端实现方式**（问题5补充）：

| 步骤 | 前端操作 | 技术实现 |
