## 一、严重问题（Critical）

### C-001：签到按钮组件硬编码用户ID

**文件**：[checkin-button/index.tsx](file:///f:/openclaw文件/与你/yu-ni-nextjs/src/components/checkin-button/index.tsx#L22)  
**问题描述**：
```typescript
const result = await checkIn('1'); // 硬编码userId为'1'
```
组件接收了`streakCount`和`checkedIn`属性，但在调用`checkIn`时硬编码了`userId`为'1'，导致所有用户签到都会记录到同一个用户ID下。

**修复建议**：
1. 从session获取真实的userId
2. 将userId作为props传入组件

**影响范围**：签到功能、好感度计算

---

### C-002：聊天消息格式不一致

**文件**：[partner.ts](file:///f:/openclaw文件/与你/yu-ni-nextjs/src/server/actions/partner.ts#L46-L58)  
**问题描述**：
```typescript
// partner.ts 中创建聊天会话时直接传入对象数组
messages: [
  { role: 'ai', content: `你好呀！我是${data.name}...`, timestamp: new Date().toISOString() },
]

// chat.ts 中创建聊天会话时使用JSON.stringify
messages: JSON.stringify([
  { role: 'user', content: message },
  { role: 'assistant', content: response.content },
])
```
`chatSession.messages`字段在Prisma schema中定义为`Json`类型，但在两个action中处理方式不一致：`partner.ts`直接传入对象数组，`chat.ts`使用`JSON.stringify`转为字符串。

**修复建议**：统一使用`JSON.stringify`或将Prisma schema中messages类型改为对象数组。

**影响范围**：聊天消息存储和读取

---

### C-003：创建伴侣时性格维度值未传递

**文件**：[create-partner/page.tsx](file:///f:/openclaw文件/与你/yu-ni-nextjs/src/app/(main)/partner/create-partner/page.tsx#L62-L66)  
**问题描述**：
```typescript
await createPartner(session.user.id, {
  name: name.trim(),
  coreType: selectedType,
  relationshipOrigin,
});
```
前端页面计算了MBTI性格维度值（`getMBTIDimensions`函数），但在调用`createPartner`时没有传递这些维度值（`extroversion`, `intuition`, `feeling`, `judging`），导致伴侣创建时这些值使用默认值5，与用户选择的性格类型不匹配。

**修复建议**：将`getMBTIDimensions(selectedType)`的结果传递给`createPartner`。

**影响范围**：伴侣性格准确性、聊天角色设定

## 二、中等问题（Major）

### M-001：导航栏首页链接处理问题

**文件**：[(main)/layout.tsx](file:///f:/openclaw文件/与你/yu-ni-nextjs/src/app/(main)/layout.tsx#L42)  
**问题描述**：
```typescript
href={`/${item.id === 'home' ? '' : item.id}`}
```
当`item.id === 'home'`时，href为`'/'`，但页面实际渲染时，`(main)`布局不会应用到根路径`/`，导致导航到首页后底部导航栏消失。

**修复建议**：将首页路由改为`/home`，或调整布局结构。

**影响范围**：用户导航体验

---

### M-002：训练评估结果使用随机数据作为fallback

**文件**：[training.ts](file:///f:/openclaw文件/与你/yu-ni-nextjs/src/server/actions/training.ts#L74-L85)  
**问题描述**：
```typescript
let result: Partial<TrainingResult> & { bestResponse?: string } = {
  score: Math.floor(Math.random() * 30) + 60,
  dimensions: {
    empathy: Math.floor(Math.random() * 40) + 50,
    // ...其他维度也是随机值
  },
  suggestions: ['继续保持', '可以尝试从不同角度思考', '多加练习会更好'],
  bestResponse: '这是一个很好的开始！',
};
```
当LLM API调用失败或返回格式解析失败时，使用随机数据作为fallback，导致评估结果不可预测，影响用户训练体验。

**修复建议**：基于用户回答内容进行简单的规则匹配评分，或使用更合理的默认值。

**影响范围**：训练评估准确性

### M-003：情绪分析引擎随机分数问题

**文件**：[emotion.ts](file:///f:/openclaw文件/与你/yu-ni-nextjs/src/server/engines/emotion.ts#L46-L48)  
**问题描述**：
```typescript
} else {
  emotionType = this.simulateBERTResult(text);
  emotionScore = Math.floor(Math.random() * 50) + 30; // 随机分数

```
对于非0级会员（付费用户），情绪分析类型使用规则匹配，但分数仍然是随机生成的，导致情绪分析结果不准确。

**修复建议**：根据情绪类型和关键词匹配程度计算合理的分数值。

**影响范围**：情绪分析准确性、决策引擎输出

### M-004：Redis使用内存Map模拟

**文件**：[redis.ts](file:///f:/openclaw文件/与你/yu-ni-nextjs/src/lib/redis.ts#L1-L23)  
**问题描述**：
```typescript
const cache = new Map<string, string>();

export const redis = {
  get: async (key: string): Promise<string | null> => {
    return cache.get(key) || null;
  },
  // ...其他方法
};
```
使用JavaScript的`Map`对象模拟Redis，服务器重启后所有缓存数据会丢失，包括聊天计数、每周模拟次数等。

**修复建议**：
1. 安装真实Redis服务并连接
2. 或使用文件存储作为持久化方案
3. 或在应用启动时从数据库初始化缓存

**影响范围**：聊天计数、模拟次数限制

## 四、代码优化建议

### O-001：统一错误处理

多个API和Server Actions使用了不同的错误处理方式，建议统一使用try-catch包装，并返回标准化的错误格式。

### O-002：类型定义集中管理

建议将通用类型定义（如`ChatMessage`, `TrainingResult`, `AbilityRadar`等）集中到`src/types/`目录下，避免在多个文件中重复定义。

### O-003：增加输入验证

在所有用户输入相关的API和Server Actions中增加输入验证（可使用zod），防止恶意输入。

### O-004：添加单元测试

建议为核心引擎（情绪分析、决策引擎、好感度计算）添加单元测试，确保逻辑正确性。
